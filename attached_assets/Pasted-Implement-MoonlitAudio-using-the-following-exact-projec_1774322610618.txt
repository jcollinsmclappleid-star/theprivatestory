4. RESPONSIBILITY OF EACH FILE
==================================================

----------------------------------
server.js
----------------------------------
- boot express app
- serve static frontend from /public
- mount routes
- apply middleware
- start server

----------------------------------
/routes/storyRoutes.js
----------------------------------
Own these endpoints:
- POST /api/plan-story
- POST /api/generate-story
- POST /api/qc-story
- POST /api/rewrite-story
- POST /api/generate-image-prompts
- POST /api/generate-images
- POST /api/generate-audio
- POST /api/generate-full-story
- POST /api/generate-variation
- POST /api/continue-story
- GET /api/story/:id

----------------------------------
/routes/userRoutes.js
----------------------------------
Own:
- GET /api/recommendations/:userId
- POST /api/update-progress

----------------------------------
/routes/libraryRoutes.js
----------------------------------
Own:
- POST /api/save-story
- GET /api/library/:userId

----------------------------------
/routes/seriesRoutes.js
----------------------------------
Own:
- GET /api/series
- GET /api/series/:id

----------------------------------
/services/plannerService.js
----------------------------------
- call planner prompt
- turn normalised intake into hidden brief
- validate output against schema

----------------------------------
/services/writerService.js
----------------------------------
- call writer prompt
- produce story json
- validate output

----------------------------------
/services/qcService.js
----------------------------------
- call QC prompt
- score story
- return pass/fail + rewrite strategy

----------------------------------
/services/rewriteService.js
----------------------------------
- apply targeted rewrite if QC fails
- preserve story identity

----------------------------------
/services/imagePromptService.js
----------------------------------
- generate cover + scene prompts from brief + story

----------------------------------
/services/imageService.js
----------------------------------
- call image API
- generate cover + scene images
- cache results

----------------------------------
/services/audioService.js
----------------------------------
- combine scenes into narration script
- select ElevenLabs voice
- generate audio
- return audio URL / file data

----------------------------------
/services/variationService.js
----------------------------------
- generate softer / darker / slower / more emotional / new ending / new setting / continue chemistry versions

----------------------------------
/services/continuationService.js
----------------------------------
- create next chapter from existing story

----------------------------------
/services/recommendationService.js
----------------------------------
- use user taste profile + saves + listens to recommend stories

----------------------------------
/services/memoryService.js
----------------------------------
- update taste profile
- track moods, endings, replays, saves, intensities

----------------------------------
/services/storageService.js
----------------------------------
- read/write json files for MVP
- abstract storage so DB can be swapped in later

----------------------------------
/prompts/*
----------------------------------
Store all prompt strings in separate files.
Do not hardcode prompts directly inside routes.

----------------------------------
/data/*
----------------------------------
Use JSON files for MVP persistence.

7. REQUEST HASH / CACHING
==================================================

Use hashRequest.js to create deterministic signature for generation requests.

Hash input:
name + mood + intensity + length + scenario + cinematic_visuals + emotional_focus + voice_feel

Before generating:
- check generatedCache.json
- if matching completed story exists, return cached version
- only regenerate if user requested variation or continuation

==================================================
8. MVP STORAGE STRATEGY
==================================================

Use JSON files first.

stories.json
- all curated + generated stories

users.json
- taste profile + saved stories + generated story ids

progress.json
- current listening progress per user/story

generatedCache.json
- request hash → story id

Later you can swap to DB without changing service logic much.

==================================================
9. ERROR HANDLING RULES
==================================================

Every route should:
- validate input
- catch API errors
- return structured error json

Error response shape:
{
  "error": true,
  "message": "Something interrupted the story generation."
}

On frontend:
- show elegant error card
- allow retry
